# 週次レポート設計 v0.6

## 1. 先に結論
今回の実装は「週次レポート」を作っていたが、ユーザーが求めていたのは「日次レポートの読み味を持つ週次レポート」だった。

ずれていたポイントは次の3つ。

- 現行設計は `TOP10` と `クラスタリング結果` を主成果物として扱っていた
- 週報本文が `hour_title` や `cluster_labels` の再掲に近く、自然言語の叙述が主役になっていなかった
- 1週間を「数字で集計する対象」として扱いすぎており、「何を進めた週だったのか」を文章で読ませる設計になっていなかった

以後の設計では、主成果物を「自然言語の週次記述」に戻す。

## 2. 再定義した目的
この機能の目的は、1週間分のPC作業ログをもとに、既存の日次レポートと同じ読み味で「何をして、何が進み、次に何をするか」が分かる文章を生成すること。

重要なのは、単なる集計ではなく振り返りとして読めること。

そのため、週次レポートの主役は以下とする。

- 週全体の自然言語による総括
- 日ごとの自然言語による小さな日報
- 必要に応じたハイライト

逆に、以下は補助情報へ格下げする。

- TOP10
- 時間集計
- クラスタリング結果
- 利用ツール名の列挙

## 3. 出力要件の見直し
### 3.1 主成果物
- 主成果物は `weekly.report.md`
- `weekly.report.md` は「読める文章」であることを最優先にする
- 文章を読むだけで、その週に何をしていたかの見取り図が掴めること

### 3.2 読み味の要件
- 文体は既存の日次レポートに寄せる
- 「アプリ名 / URL / window_title」の羅列を本文の主役にしない
- 各段落は「行動 + 対象 + 目的 + 進捗」で書く
- 可能なら「流れ」「判断」「結果」「次のアクション」まで含める
- 1時間ごとの観測事実は、本文ではなく根拠として利用する

### 3.3 禁止事項
- 本文の見出し直下が表だけになること
- `Slack / Chrome / Notion / Cursor` のようなツール名列挙が主たる説明になること
- `cluster_labels` や `canonical_task_name` をそのまま本文へ貼ること
- 「TOP10が正しいから本文も十分」とみなすこと

## 4. 週次レポートの正しいイメージ
### 4.1 望ましい出力イメージ
週報は、次のような構造を持つ。

1. 週のタイトル
2. 今週の総括
3. 今週のハイライト
4. 日別の記述
5. 補足の集計情報

読み順の中心は 2, 3, 4 である。

### 4.2 日次レポートとの対応
既存の日次レポートでは、自然言語らしさは主に以下から生まれていた。

- `daily_title`
- `daily_summary`
- `highlights`
- `hour_summary_enriched`

つまり、日次レポートは「観測された画面」ではなく「その時間の意味」まで一段抽象化していた。

週次でも同じで、主入力は raw な `hour_title` ではなく、「その日が何の日だったか」という日次レベルの意味づけに置くべきである。

## 5. 要件（更新後）
### 5.1 機能要件
- 毎週月曜日に、先週（月〜日）の週次レポートをMarkdownで自動生成する
- 週次レポートは、週全体の自然言語総括を必ず含む
- 週次レポートは、各日の自然言語サマリを必ず含む
- TOP10や集計表は補助情報として掲載してよいが、本文の主役にしない
- 部分欠損があっても、読める文章を優先して週報生成を継続する
- Notion同期は週次レポート本文をそのまま使って行う

### 5.2 品質要件
- 本文だけ読めば、その週に何を進めたかが分かる
- 各日について、ツール列挙ではなく「その日何をしていたか」の自然言語要約が最低1段落ある
- LLM失敗時でも、本文がツール名の羅列に崩れない
- 集計が壊れても、週報本文はできるだけ生成される

### 5.3 保存/保持要件
- `out/` 保持期間は 14日
- `weekly/days/` は 20日保持
- `weekly/weeks/` は 20日保持

### 5.4 モデル要件
- `hourly-llm` / `daily-llm` / `hour-enrich-llm` / `weekly-llm` は `gpt-5-nano` 固定
- ただし、週報本文の品質は `weekly-llm` 単体に依存させず、日次側の意味づけ済みデータを必ず利用する

## 6. データモデルの見直し
### 6.1 問題だった点
現行の `weekly/days/YYYY-MM-DD.hourly.json` は、週次集計に必要な最低限の構造しか持っていない。

そのため週次側では、

- `hour_title`
- `hour_summary`
- `cluster_labels`

を直接クラスタリングし、そのまま本文へ近い場所で再利用してしまう。

これが「自然言語ではなく、ツール列挙に見える」主因である。

### 6.2 新しい日次スナップショット要件
週次の主入力は「1時間単位の記録」ではなく「1日単位の意味づけ済み要約」にする。

そのため `build_day_snapshot(date)` が保存すべき内容を拡張する。

保存先:
- `EVERYTIME-LOG/weekly/days/YYYY-MM-DD.hourly.json`

必須項目:
- `date`
- `source_run_id`
- `generated_at`
- `model`
- `status` (`complete` / `incomplete`)
- `incomplete_reason`
- `total_active_sec_est`
- `hours[]`
  - `hour_start_ts`
  - `hour_end_ts`
  - `active_sec_est`
  - `hour_title`
  - `hour_summary`
  - `hour_title_enriched`
  - `hour_summary_enriched`
  - `cluster_labels[]`
  - `confidence`
- `daily`
  - `daily_title`
  - `daily_summary`
  - `daily_detail`
  - `highlights[]`
  - `confidence`

原則:
- 週次本文の主入力は `daily.*` と `hour_summary_enriched`
- `hour_title` と `cluster_labels` は補助情報

## 7. 週次本文の生成設計
### 7.1 基本方針
週報本文は二段階で組み立てる。

1. 各日の「日報らしい要約」を確定する
2. その7日分をもとに、週全体の総括を生成する

つまり、週報を raw log から直接書かない。

### 7.2 日別セクションの作り方
各日について、以下の優先順で文章を構成する。

1. `daily.daily_detail`
2. `daily.daily_summary`
3. `hour_summary_enriched` の束から生成した日別短文
4. `hour_summary` の束から生成した日別短文

日別セクションの出力形式:

- 見出し: `### YYYY-MM-DD`
- 1段落目: その日何をしていたか
- 2段落目: 進捗 / 判断 / 変化 / 次アクション
- 必要に応じて `highlights` を3件まで箇条書き

重要:
- 日別セクションに表は不要
- `主な作業: Slack` のような断片列挙をそのまま出さない

### 7.3 週全体の総括の作り方
週全体の総括は、7日分の `daily_title` / `daily_summary` / `daily_detail` / `highlights` を入力にして生成する。

入力には補助的に以下を含めてよい。

- 日別の推定稼働時間
- 週全体の主要テーマ
- クラスタリング結果

ただし、週全体の総括はクラスタリング結果の読み上げではない。

総括に求める内容:
- 今週の主目的
- 何が進んだか
- どこで判断や方針転換があったか
- 次に何へつながるか

出力は次の構造を推奨する。

- `weekly_title`
- `weekly_summary`
- `weekly_detail`
- `highlights[]`

ここで `weekly_detail` を新設する。

### 7.4 週報 Markdown の構成
推奨構成:

1. `# 週次レポート YYYY-MM-DD - YYYY-MM-DD`
2. `## 今週の総括`
   - `weekly_title`
   - `weekly_summary`
   - `weekly_detail`
3. `## 今週のハイライト`
4. `## 日別ログ`
   - `### 月`
   - `### 火`
   - ...
5. `## 補足`
   - 推定総作業時間
   - TOP10
   - データ欠損メモ

ここでの原則:
- `補足` は文末
- TOP10 は本文の前に置かない
- 自然言語セクションを先に読ませる

## 8. 集計・クラスタリングの役割変更
### 8.1 現行設計の問題
現行設計では、クラスタリング結果が本文の骨格に近すぎる。

その結果:
- クラスタ名の質が悪いと本文全体が崩れる
- LLMの未割当やJSON不整合が、そのまま週報生成停止要因になる
- フォールバック時にツール名列挙が露出する

### 8.2 新しい位置づけ
クラスタリングは本文生成の主系ではなく、補助情報とする。

クラスタリングの用途:
- TOP10算出
- 週の主要テーマ抽出の補助
- 補足セクションへの掲載

クラスタリングの非用途:
- 本文そのものの骨子
- 日別セクションの直接生成

### 8.3 失敗時の扱い
クラスタリングに失敗しても週報本文は生成する。

優先順位:
1. 週報本文を出す
2. TOP10を出す
3. クラスタリングをきれいに整える

失敗時のフォールバック:
- 本文: `daily.*` を使って継続生成
- TOP10: 省略または簡易集計
- pending: `task_clustering` として再試行対象に残す

## 9. 部分欠損時の設計
### 9.1 方針
部分欠損は「週報を止める理由」ではなく「品質注記を付ける理由」とする。

### 9.2 日次欠損の扱い
- `daily` があれば、その日を usable とみなす
- `daily` が無くても `hour_summary_enriched` が複数あれば usable
- `hour_summary` しか無くても、短文生成で日別記述を作れるなら usable
- 何も無ければ、その日だけ「記録不足」と明示する

### 9.3 週報への反映
- 週報は生成する
- `## 注意` に欠損日と理由を書く
- 欠損日も、可能な範囲で自然言語の短い説明を付ける
- pending は残して再補完対象にする

## 10. LLMフローの見直し
### 10.1 日次側
日次側では、週次に必要な意味づけを確実に保存する。

必須:
- `hourly-llm`
- `daily-llm`

推奨:
- `hour-enrich-llm`

### 10.2 週次側
週次側では2種類のLLMを分ける。

1. `weekly-summary-llm`
   - 7日分の日別要約から週の総括を書く
2. `weekly-day-polish-llm` または同等の整形
   - 欠損日や粗い日次要約を、週報用の自然な短文へ整える

ただし、週次側で raw hourly ログを大きく再解釈する設計にはしない。

### 10.3 タイムアウトと再試行
週次側の LLM 呼び出しは本文の可用性を優先する。

原則:
- 週次側の LLM タイムアウトは短めにする
- 長時間待つより、フォールバックして本文を出す
- 失敗時は pending に残しつつ今回の週報は生成する

## 11. Notion同期
週次レポート生成完了後、`weekly.report.md` を Notion へ同期する。

同期対象:
- `weekly.report.md`

同期原則:
- Notion に送るのは統計ではなく本文
- ページタイトルは `週次レポート YYYY-MM-DD - YYYY-MM-DD`
- 既存の日次DB/プロパティを流用してよい
- 同期失敗時は `stage=notion_sync` で pending に残す

## 12. CLIと運用
### 12.1 CLI
- `everlog weekly-run`
- `everlog weekly-run --week-start YYYY-MM-DD`
- `everlog weekly-run --retry-pending-only`
- `everlog weekly-run --force`

### 12.2 `daily-run` の責務
`daily-run` は廃止しない。

理由:
- 週次本文の品質は日次意味づけの品質に依存するため
- `daily-llm` と `hour-enrich-llm` を事前に揃える価値が高いから

`daily-run` の役割:
- pending / yesterday / today の日次スナップショットを揃える
- 必要なら週次 pending の再実行を起動する

## 13. 実装方針
### Phase 1
- `build_day_snapshot()` に `daily` と `hour_summary_enriched` を保存する
- 週報本文を `daily.*` 主入力へ切り替える
- 日別セクションを自然言語中心へ変更する
- TOP10 を補足セクションへ移動する

### Phase 2
- `weekly_detail` を生成する LLM schema を追加する
- 部分欠損日の短文整形を追加する
- Notion 同期の週次本文を安定化する

### Phase 3
- クラスタリング品質改善
- 件数が多い週の分割処理
- 週報本文の品質メトリクス導入

## 14. 受け入れ基準
- `weekly.report.md` を開いたとき、最初に自然言語の総括が読める
- 各日ごとに、ツール列挙ではない自然言語の記述がある
- TOP10 や表は存在してもよいが、本文の後ろに置かれている
- クラスタリングに失敗しても週報本文は生成される
- 部分欠損があっても週報本文は生成され、欠損は注記として扱われる
- 週報本文で `Slack / Chrome / Notion / Cursor` の羅列が主説明になっていない
- 週報本文が、既存の日次レポートの読み味に近い

## 15. 今回の判断
今回の不具合は「LLMの精度不足」だけではない。設計が、文章生成より集計を主目的にしていたことが本質的な原因だった。

したがって、今後の修正方針は次の通り。

- 週報を「集計物」ではなく「週次の日報」として扱う
- 日次で作っていた意味づけ済みテキストを週次の主入力にする
- クラスタリングは本文の裏方へ下げる
- 欠損や LLM 失敗があっても、読める文章を先に出す
